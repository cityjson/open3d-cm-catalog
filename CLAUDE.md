# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

The marketing/documentation website for the **Open 3D City Model Catalog** — a project that defines a
[STAC](https://stacspec.org/) extension for 3D city models (CityJSON, CityGML) and maintains a central,
community-curated registry of 3D city models. The companion data/registry repo lives at
<https://github.com/cityjson/city3d-stac-registry>; this repo is only the website that introduces the
project and drives contributions. The catalog data is contributed by anyone via GitHub PRs.

## Commands

Package manager is **pnpm** (do not use npm/yarn — there is a `pnpm-lock.yaml`).

| Task                      | Command                                                                            |
| ------------------------- | ---------------------------------------------------------------------------------- |
| Dev server                | `pnpm dev`                                                                         |
| Production build (static) | `pnpm build` → `dist/`                                                             |
| Preview built site        | `pnpm preview`                                                                     |
| Type-check `.astro`/`.ts` | `pnpm check` (`astro check`)                                                       |
| Lint                      | `pnpm lint` / autofix `pnpm lint:fix`                                              |
| Format                    | `pnpm format` (writes) / `pnpm format:check`                                       |
| Lint or format one file   | `pnpm exec oxlint src/path/file.tsx` · `pnpm exec oxfmt --write src/path/file.tsx` |
| Everything CI runs        | `pnpm validate` (format:check + lint + check)                                      |

There is no test runner configured yet.

## Architecture

**Static-site generator: Astro 6.** Output is fully static (`output: "static"`). Astro embeds Vite, so
there is no standalone Vite config — Vite plugins (e.g. Tailwind) are registered under `vite.plugins` in
`astro.config.mjs`, not a `vite.config.*` file.

**React 19 for interactivity, via Astro islands.** `.astro` files render to static HTML at build time and
ship zero JS. Anything interactive must be a React `.tsx` component mounted in an `.astro` file with a
`client:*` directive (e.g. `<CatalogCount client:load />`). Without a `client:` directive a React component
renders statically and is _not_ hydrated. See `src/components/CatalogCount.tsx` + its use in
`src/pages/index.astro` for the canonical pattern.

**Tailwind v4 + shadcn/ui.** Tailwind v4 is CSS-first: there is **no `tailwind.config.js`**. The design
tokens (colors, fonts, radii) live as CSS variables and `@theme` blocks in `src/styles/global.css` — change
the look by editing that file. shadcn primitives live in `src/components/ui/` and are generated by the
shadcn CLI (style preset `radix-nova`, configured in `components.json`). Add new ones with
`pnpm dlx shadcn@latest add <component>`. Prefer regenerating/extending via the CLI over hand-editing files
in `src/components/ui/`. `cn()` (clsx + tailwind-merge) lives in `src/lib/utils.ts`.

**Path alias:** `@/*` → `src/*` (defined in `tsconfig.json`, mirrored in `components.json`).

## Linting/formatting toolchain — read this

Lint and format use the **Oxc** tools — **Oxlint** (`.oxlintrc.json`) and **Oxfmt** (`.oxfmtrc.json`) — the
same engines that power "Vite+". They were chosen over the full Vite+ toolchain because Vite+ is
alpha/commercial and its build entry point conflicts with Astro's own CLI; the underlying Oxc binaries give
the same speed and rules while staying free for outside contributors.

Important limitations to keep in mind:

- **Oxlint and Oxfmt only process JS/TS/JSX/TSX.** They do **not** parse `.astro` files. `.astro` correctness
  is covered by `astro check` (run via `pnpm check`); there is currently no autoformatter for `.astro`, so
  keep `.astro` markup tidy by hand.
- **The `src/components/ui/` directory is excluded** from both Oxlint and Oxfmt so generated shadcn
  components don't drift from upstream formatting. Don't be surprised when format/lint skip them.
- `react/react-in-jsx-scope` is disabled on purpose — the project uses the modern JSX transform
  (`jsx: "react-jsx"`), so importing `React` is unnecessary.

## Conventions

- Components needing browser state/events → React `.tsx` with a `client:` directive. Everything else → `.astro`.
- New shadcn components via the CLI, not copy-paste.
- Run `pnpm validate` before considering a change done.
